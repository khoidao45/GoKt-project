OutboxProcessor
│
├─ BEGIN TRANSACTION
├─ SELECT * FROM OutboxEvents
│    WHERE Status='Pending'
│    ORDER BY CreatedAt
│    LIMIT 20
│    FOR UPDATE SKIP LOCKED        ← multiple instances safe
│
├─ For each event:
│   ├─ KafkaPublisher.PublishRawAsync(type, key, payload)
│   │   ├─ success → event.MarkProcessed()
│   │   └─ fail    → event.IncrementRetry()
│   │                  if RetryCount >= 5 → event.MarkFailed()
│
├─ db.SaveChangesAsync()
└─ COMMIT TRANSACTION

                   (Kafka is up)
MatchingWorker / KafkaMatchingConsumer
│
└─ Consume "ride.requested"
    └─ IMatchingService.StartMatchingAsync(rideRequestId)
        └─ AutoMatchingStrategy → wave-based driver matching
Key design decisions
Decision	Reason
Same SaveChangesAsync for ride + outbox	Guarantees atomicity — if DB commit fails, no orphan Kafka message is produced
FOR UPDATE SKIP LOCKED	Multiple OutboxProcessor instances (API + MatchingWorker) never process the same row
MessageKey stored on OutboxEvent	Kafka partition routing is stable on replay; the consumer sees the same key on retry
PublishRawAsync	Payload is already serialized when stored; avoids double-serialization
EnableIdempotence = true on producer	Kafka producer-side deduplication within a session (network retries don't produce duplicates)
LastError column on OutboxEvent	Failed events are self-documenting in the DB — no grep-through-logs needed
IDesignTimeDbContextFactory	Migrations work without needing Redis/Kafka running
Max 5 retries → Failed status	Events that exceed retries are visible for alerting rather than silently dropped
